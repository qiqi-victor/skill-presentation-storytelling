# Presentation Storytelling Skill

> 角色: 演讲叙事策划 🎯 | 版本: v3.6 | 更新: 2026-04-06 17:28 | 维护者: 🐼🦐子
> 核心变化: Step 0 新增「演讲上下文捕获」必填节（场合/听众/目的/时长），缺失时必须询问
> 重大修复: 场合+听众+目的是每一次演讲的必备信息，缺失时必须询问，不执行叙事设计
> 名称更新: narrative-design → presentation-storytelling（更面向终端用户）
> 定位: PPT 专项三角色之一（策划→编剧→美工）
> 核心变化: Layer 1 拆分为 1A（通用演讲）+ 1B（新技术落地），非技术场景也能用；PM 审查后显性化五情境（A/B/D/E）+ E 场景支持
> 产出链路: 材料理解报告 → 叙事大纲（含页数预算）→ 故事脚本（逐页主旨）→ 交编剧执行

## 四层架构说明

```
┌─────────────────────────────────────────────┐
│  Layer 3: Personalization Layer (个性化)   │
│  - 风格选项（叙事风格/语言风格/情感强度）    │
│  - 用户主导调整，Agent 提供预设和建议        │
├─────────────────────────────────────────────┤
│  Layer 2: Domain Layer (行业特定)          │
│  - 行业知识（术语/痛点/决策框架/演讲案例）  │
│  - 通过 Deep Research 动态获取              │
│  - 所有行业统一走 Deep Research               │
├─────────────────────────────────────────────┤
│  Layer 1B: Tech Landing (新技术落地专项)   │
│  - 价值判断 / 认知诊断 / 技术悖论 / 场景价值论 │
│  - 仅在"新技术落地"情境时激活                │
├─────────────────────────────────────────────┤
│  Layer 1A: Universal (通用演讲方法论)      │
│  - 普适叙事框架（SCQA/STAR/金字塔/英雄之旅）│
│  - 六问 / 案例库 / 诚实原则                 │
│  - 所有行业、所有听众、所有 PPT 场景适用     │
└─────────────────────────────────────────────┘
```

**激活规则**:
- 所有 PPT 场景 → 必用 Layer 1A
- 新技术落地/AI/区块链/云计算等场景 → 加用 Layer 1B
- 非技术场景（公司年会、路演、述职）→ 只用 Layer 1A

## 流程控制: Skip / Reduce / Adapt

### 判断逻辑

Step 0 执行前，先扫描用户输入：

| 用户输入 | 动作 | 说明 |
|---------|------|------|
| 用户已有叙事大纲 | **Skip** Step 1-5 | 直接跳到故事脚本或 Completeness 评分 |
| 用户已有故事脚本 | **Skip** 全部 Workflow | 直接输出评分 + 改进建议 |
| 用户说"简版" | **Reduce** L1B+L2+L3 | 只用 L1A 普适方法论 |
| 用户说"行业版" | **Reduce** L1B+L3 | 加 L2，跳过技术专项和风格适配 |
| 用户说"完整版" | 全部 Step | L1+L2+L3 全部执行 |
| 用户指定行业 | **Adapt** Deep Research | 按指定行业搜索 |
| 用户指定页数 | **Adapt** 故事脚本 | 按页数控制节奏 |
| **A 改造型**（多份散落材料） | **Adapt** Step 0.6 | 从"理解材料"开始，逐份分析+交叉分析 |
| **D 翻新型**（他人/旧 PPT） | **Adapt** Step 0.6 | 从"理解材料"开始，重点在重构叙事 |
| **B 创作型**（从零开始） | **Adapt** 入口 | 无需 Step 0.6，直接 Step 1（明确约束+叙事设计） |
| **E 局部修改**（改几页） | **Adapt** 定向重做 | 识别需修改的页面，定向重做，不破坏整体叙事 |
| 用户已有 PPT 素材 | **Adapt** 入口 | 从"理解材料"开始，不从零设计 |

### 五情境识别与处理路径

> 本 Skill 显性覆盖 A/D/B/E 四种情境，C 美化（纯排版）非本 Skill 目标。

| 情境 | 识别信号 | 执行路径 | Step 0.6 |
|------|---------|---------|----------|
| **A 改造型** | 多份散落材料（PDF/文档/链接） | Step 0.6→1→2→3→4→5→6 | ✅ 必做 |
| **D 翻新型** | 他人 PPT / 旧 PPT → 我的演讲 | Step 0.6→1→2→3→4→5→6 | ✅ 必做 |
| **B 创作型** | 有主题/想法，无材料 | Step 1（明确约束）→5（直接叙事设计） | ❌ 跳过 |
| **E 局部修改** | "只改这几页" / "就换这几张" | 定向重做指定页面→美工局部替换 | ❌ 跳过 |
| **C 美化型** | "帮我美化/排版，内容不动" | 建议转美工 Skill（策划价值有限） | ❌ 不适用 |

### 速查

```
用户有大纲 → Skip to 故事脚本
用户有脚本 → Skip to 评分
用户说简版 → Reduce L1A only
用户说行业版 → Reduce L1A+L2
用户说完整版 → 全部执行
A/D 用户提供多份材料 → Step 0.6 材料理解（必做，等确认）
B 用户有主题无材料 → Step 1 直接叙事设计
E 用户说"只改这几页" → 定向重做指定页面
C 用户只要求美化 → 建议转美工 Skill
用户指定页数 → Adapt 故事脚本按页数展开
```

## Layer 1A: 通用演讲方法论（所有 PPT 场景必用）

> **核心价值**: 这是任何演讲都适用的基础方法论，不局限于任何特定主题或行业。

### 1A.1 诚实原则

```
❌ 禁止: "一句话全自动" / "我们的技术最强" / 回避不确定性
✅ 要求: 诚实说明边界 / 明确告知"需要人在回路" / 用数据说话
```

---

### 1A.2 普适叙事框架

- **SCQA**（说服型）：情境→冲突→问题→答案
- **STAR**（案例型）：情境→任务→行动→结果
- **金字塔原理**（逻辑型）：结论→支撑点→细节
- **英雄之旅**（励志型）：召唤→启程→试炼→转化→归来

---

### 1A.3 Story Challenge 六问

```
1. 听众是谁？他们关心什么？
2. 核心信息是什么？一句话能说清吗？
3. 开头能抓住注意力吗？前 30 秒决定生死
4. 逻辑线清晰吗？听众能跟上吗？
5. 结尾有行动召唤吗？听众该做什么？
6. 有什么是可以删的吗？少即是多
```

### 1A.4 经典演讲案例库

#### Jobs iPhone 2007 — "三合一"戏剧揭示
- 先拆三个产品（iPod + 手机 + 上网设备）→ 揭示"三合一就是 iPhone"
- **可复用**: 先拆后合，制造惊喜峰值

#### Jobs Stanford 2005 — 三故事递进
- 串连点滴→爱与失去→死亡，从个人到普遍
- **可复用**: 名言收尾（"Stay Hungry. Stay Foolish."）

#### 5 大叙事结构速查

| 结构 | 模式 | 适用场景 | 代表演讲 |
|------|------|---------|---------|
| **三选一揭示** | 先拆后合，制造惊喜 | 产品发布 | Jobs iPhone 2007 |
| **三故事递进** | 从个人到普遍，细节饱和 | 激励演讲 | Jobs Stanford 2005 |
| **演示驱动** | 让产品自己说话 | 技术演示 | Musk Cybertruck 2019 |
| **冲突叙事** | 把质疑变成流量 | 挑战现状 | 雷军 SU7 2024 |
| **人格驱动** | 允许不完美成为连接器 | 个人品牌 | 罗永浩 坚果Pro 2017 |

## Layer 1B: 新技术落地专项（仅技术场景激活）

> **适用条件**: 演讲主题涉及新技术落地（AI/区块链/云计算/IoT/自动驾驶等）。
> **非技术场景跳过本节**（公司年会，投资人路演、述职等）。

### 1B.1 价值判断框架

| 层级 | 定义 | 叙事策略 |
|------|------|---------|
| **确定能做的** | 高确定性、成熟应用 | "为您完成" |
| **概率性擅长的** | 需要人在回路 | "在您指导下完成" |
| **当前做不到的** | 技术边界外 | 诚实说明 |

---

### 1B.2 认知诊断三阶段

| 阶段 | 特征 | 应对策略 |
|------|------|---------|
| **概念验证期** | 刚接触，充满好奇和怀疑 | 提供成功案例，建立信任 |
| **试点期** | 小范围尝试，关注 ROI | 聚焦具体场景，给出明确价值 |
| **规模化期** | 大范围推广，关注效率 | 强调组织变革、流程融合 |

---

### 1B.3 两个技术悖论

**悖论 1: 时间悖论** — 想自己构建能力，但迭代太快跟不上。普适结论：使用成熟服务 > 自建能力

**悖论 2: 空间悖论** — 想做一个平台服务所有场景。普适结论：场景聚焦 > 平台思维

---

### 1B.4 场景价值论

**核心原则**: 技术是通用能力，场景是价值载体

| 问题 | B 端回答 | C 端回答 |
|------|---------|---------|
| **为什么场景是根本？** | 技术不创造价值，场景创造价值。65% 失败因场景选错 | 用户不为技术买单，为场景体验买单 |
| **为什么只有技术不够？** | 端到端四层：技术层→应用层→数据层→体验层 | 用户要"省心"，不是"技术参数" |
| **什么是真正的产品？** | 能端到端解决用户问题的方案 | 让用户无感使用的产品 |

## Layer 2: Deep Research 框架

### 2.1 搜索条件配置

**必需（3 类）**：行业背景 / 听众画像 / 行业痛点
**可选（2 类）**：经典演讲 / 叙事风格

### 2.2 Researcher 交互协议

**调用方式**: `sessions_spawn(agentId: "researcher", task: "...")`

**Task 构造模板**:
```
搜索以下内容，返回结构化结果：
1. {行业} 行业概况（市场规模、增长率、头部玩家）
2. {行业} {听众角色} 核心痛点（至少 5 条）
3. {行业} 决策流程（谁决策、什么因素影响）
4. {行业} 经典演讲/案例（如有）
搜索来源：kimi_search + web_search，中英文都搜。
```

**返回格式**: 非固定，但需覆盖 3 必需维度（行业背景/听众画像/行业痛点）

**校验失败处理**:
- 缺少 3 必需维度中的任何一个 → 重新搜索该维度
- 搜索延迟 > 30s → 跳过 L2，用 L1 普适方法论兜底
- 返回内容无行业"内行话" → 标记为低质量，降权使用

### 2.3 验证标准

- 是否包含行业"内行话"
- 是否覆盖 3 必需维度
- 搜索延迟 < 20s

## Layer 3: 个性化风格选项

### 风格维度

| 维度 | 选项 | 影响 |
|------|------|------|
| **叙事风格** | 激情型 / 理性型 / 幽默型 / 故事型 | 整体基调 |
| **语言风格** | 正式 / 口语化 / 技术术语多 / 通俗易懂 | 用词选择 |
| **信息密度** | 精简 / 适中 / 详细 | 内容深度 |
| **情感强度** | 冷静 / 温和 / 激昂 | 表达力度 |
| **互动程度** | 单向输出 / 适度互动 / 高度互动 | 听众参与 |

### 风格预设

- **激情型**：雷军/罗永浩 — 口语化、激昂、高互动
- **理性型**：李开复/吴恩达 — 正式、冷静、技术术语多
- **故事型**：Jobs Stanford — 口语化、温和、适度互动

## Workflow v3.5

### Step 0: 演讲上下文捕获 → 流程判断

> **重要性**：场合/听众/目的是每一次演讲的必备信息。这些信息直接影响叙事策略、语言风格、内容深度和行动号召。缺失这些信息的执行是没有意义的。

---

#### Step 0.1: 演讲上下文捕获（必做）

**用户输入扫描**：用户说「帮我做演讲」「帮我重构」「帮我写故事线」时，首先提取：

| 信息 | 说明 | 缺失时的处理 |
|------|------|-------------|
| **场合** | 去哪里讲？（GDPS大会/客户汇报/内部评审/产品发布会） | ⚠️ **必填，必须询问** |
| **听众** | 谁来听？（人群+规模：300位高管/技术团队/投资人） | ⚠️ **必填，必须询问** |
| **目的** | 想达到什么？（说服/汇报/拿预算/教育/激励/拿单） | ⚠️ **必填，必须询问** |
| **时长** | 多少分钟？（30分钟/5分钟闪电演讲） | 可默认（30分钟），但应告知用户 |
| **演讲者身份** | 我是谁？（产品总监/CEO/一线工程师） | 可选，影响叙事人称 |

**信息捕获决策树**：

```
用户说「帮我做/重构演讲」
│
├─ Step 0.1.1: 场合（必填）
│   ├─ 用户输入里有场合 → 记录
│   └─ 用户输入里没有场合 → 输出询问：「这个演讲是在哪里讲？」（暂停，等回复）
│
├─ Step 0.1.2: 听众（必填）
│   ├─ 用户输入里有听众 → 记录
│   └─ 用户输入里没有听众 → 输出询问：「听众主要是哪类人？大概多少人？」（暂停，等回复）
│
├─ Step 0.1.3: 目的（必填）
│   ├─ 用户输入里有目的 → 记录
│   └─ 用户输入里没有目的 → 输出询问：「你希望通过这次演讲达到什么目的？」（暂停，等回复）
│
└─ Step 0.1.4: 时长（可选，默认30分钟）
    ├─ 用户输入里有时长 → 记录
    └─ 用户输入里没有时长 → 告知用户：「默认30分钟，可以吗？」（可跳过继续）
```

**Step 0.1 输出**：

```markdown
## 演讲上下文

| 项目 | 内容 | 来源 |
|------|------|------|
| 场合 | {去XX大会/客户XX汇报} | 用户输入/询问补充 |
| 听众 | {300位企业高管/技术团队} | 用户输入/询问补充 |
| 目的 | {说服客户买单/向领导申请预算} | 用户输入/询问补充 |
| 时长 | {30分钟} | 用户输入/默认 |
| 演讲者 | {产品总监贾安亚} | 用户输入（可选） |

**确认**：以上信息是否准确？（如需调整请告诉我）
```

---

#### Step 0.2: 流程判断（在上下文捕获完成后执行）

```
输入判断:
├─ 用户已有产出物？
│   ├─ 已有叙事大纲 → Skip to Step 5（故事脚本）
│   ├─ 已有故事脚本 → Skip to Step 6（评分）
│   └─ 无 → 正常流程
│
├─ 用户提供多份材料？（A/D 场景）
│   ├─ 是 → Step 0.6 材料理解（必做，等确认后继续）
│   └─ 否 → 跳过 Step 0.6
│
├─ 是否新技术落地场景？（AI/区块链/云计算/IoT/自动驾驶等）
│   ├─ 是 → 激活 Layer 1B（认知诊断 + 第一原理 + 场景价值论）
│   └─ 否 → 跳过 Layer 1B（只用 Layer 1A 通用方法论）
│
├─ 用户要求版本？
│   ├─ "简版" → Reduce: L1A only（跳过 L1B/L2/L3）
│   ├─ "行业版" → Reduce: L1A+L2（跳过 L1B/L3）
│   └─ "完整版" → 全部 Layer
│
├─ Layer 判断
│   ├─ 有行业需求 → L2 Deep Research（**使用 Step 0.1 的听众/场合信息**）
│   └─ 有风格需求 → L3 风格适配（**使用 Step 0.1 的场合/目的信息**）
│
└─ 输出: 执行路径 + 需要的 Layer（L1A 必用，L1B/L2/L3 按条件激活）
```

**Step 0.1 + Step 0.2 组合输出示例**：

> ## 演讲上下文
> 
> | 项目 | 内容 |
> |------|------|
> | 场合 | GDPS 2026 行业大会 |
> | 听众 | 300 位企业中高管理层 |
> | 目的 | 建立信任，推动客户合作 |
> | 时长 | 30 分钟 |
> 
> ## 执行路径
> 
> 场景识别：**D 翻新**（用户提供了一份 PPT，说"帮我变成我自己的"）
> 激活 Layer：L1A + L1B（新技术落地）+ L2（Deep Research）+ L3（风格适配）
> 版本：完整版
> 
> 下一步：Step 0.6 材料理解（读取 PPT，分析原叙事线，识别重构机会）

### Step 0.5: 页数预算分配（Layer 1A）

> 演讲时长 → PPT 页数 → 各章节页数配额 → 每页内容深度

#### 时长→页数 经验公式

| 演讲时长 | 总页数 | 节奏 |
|---------|--------|------|
| 10 min | 10-12 页 | 快节奏，每页 45-60s |
| 15 min | 15-18 页 | 标准节奏，每页 50-60s |
| 20 min | 20-25 页 | 从容节奏，每页 50-60s |
| 30 min | 28-35 页 | 深度节奏，每页 50-65s |
| 45 min | 40-50 页 | 含互动/QA，每页 50-65s |

**经验法则**: 1 页 ≈ 1 分钟（含过渡和停顿）

#### 章节页数配额

> 核心信息所在的章节给最多页数，开头结尾精简，过渡压缩。

**标准 6 章结构的页数比例**（以 30 页为例）：

| 章节 | 页数占比 | 30 页分配 | 15 页分配 | 说明 |
|------|---------|----------|----------|------|
| **第一章：认知对齐** | 10-15% | 3-4 页 | 2 页 | 快速建立基调，不宜长 |
| **第二章：问题诊断** | 10-15% | 3-4 页 | 2 页 | 悖论框架 2-3 页足够 |
| **第三章：第一原理** | 20-25% | 6-7 页 | 3-4 页 | **核心章节**，给最多空间 |
| **第四章：场景穿透** | 20-25% | 6-7 页 | 3-4 页 | **核心章节**，每个场景 2 页 |
| **第五章：诚实边界** | 10% | 3 页 | 1-2 页 | 能/不能/信任，精简有力 |
| **第六章：结尾号召** | 10% | 3 页 | 1-2 页 | 重申+号召+收尾 |
| **过渡页** | 5-10% | 2-3 页 | 1 页 | 尽量压缩，一句话过渡 |

#### 页数压缩策略

当总页数不足时，按优先级砍内容：

```
优先级 1（可砍）: 过渡页 → 合并为一句话
优先级 2（可砍）: 第五章诚实边界 → 压缩到 1 页
优先级 3（可砍）: 场景数量 → 3 个砍到 2 个
优先级 4（慎重）: 第一原理 → 保留核心洞察，删除类比/架构图
❌ 不可砍: 开头钩子 + 核心信息 + 结尾号召
```

#### 页数展开策略

当总页数充裕时，按优先级加内容：

```
优先级 1: 场景数量 → 3 个加到 4-5 个
优先级 2: 第一原理 → 加类比/架构图/数据可视化
优先级 3: 诚实边界 → 加具体 case
优先级 4: 开头 → 加第二个钩子（数据震撼）
```

#### 故事脚本的页数标注

每个页面角色标注预计时长：

```
### P5 — 悖论引入（60s）
企业 AI 落地的第一个坑叫"时间悖论"...
```

**时长分配原则**:
- 钩子页：30-45s（快速抓住注意力）
- 数据震撼页：30-45s（数字说话，不用展开）
- 核心洞察页：60-90s（需要听众消化）
- 场景页：60-90s（故事需要时间铺陈）
- 行动号召页：30-45s（简洁有力）
- 过渡页：15-20s（一句话带过）

### Step 0.6: 材料理解（A/D 场景必做，B 跳过）

> A 改造型 + D 翻新型必做，B 创作型跳过。
> **触发条件**：用户提供 ≥1 份材料文件/大段文本，且需要从中提炼演讲内容。
> - A 改造型（多份散落材料）：必须做，做完整的逐份+交叉分析
> - D 翻新型（单份/多份他人 PPT）：必须做，重点在识别原叙事线+重构机会
> - E 局部修改：跳过，不进 Step 0.6

> 适用于用户提供了多份散落材料（PDF/文档/链接）的情况。B/C/E 场景跳过。

**核心原则**: **先理解材料本身，再看怎么用。** 不带活动滤镜，不预判取舍，不预设主线。
像厨师拿到食材——先摸清每样食材的特点，再看菜谱决定怎么做。

**适用条件**: 用户输入包含 ≥1 份材料文件或大段文本，且需要从中提炼演讲内容。

**执行流程**:

```
输入: 用户提供的材料文件（活动信息已在 Step 0.5 采集，但此处不按活动过滤）
│
├─ Layer A: 逐份客观理解（无偏见地看材料）
│   ├─ 材料 i 的主题是什么（一句话）
│   ├─ 核心信息/观点有哪些（3-5 句话概括）
│   ├─ 关键数据/论据/案例（直接引用原文）
│   ├─ 材料的独特价值（这份材料里有什么是别的材料没有的）
│   └─ 信息完整性（是否需要补充、数据是否过时）
│
├─ Layer B: 交叉分析（材料之间的关系）
│   ├─ 材料之间的关联（互补 / 重叠 / 冲突）
│   ├─ 各材料的独特价值点（避免重复利用）
│   └─ 组合可能性（有哪些叙事方向可以从这些材料中产生）
│
└─ Layer C: 初步关联标注（为 Step 1 铺路，不做取舍）
    ├─ 哪些信息"可能"与活动听众相关（标注，不筛选）
    ├─ 材料之间的信息缺口（有没有需要搜索补充的）
    └─ 输出: 材料理解报告 → 等用户确认
```

**⚠️ 不做的事（留给 Step 1）**:
- ❌ 不按听众筛选材料内容
- ❌ 不决定"用哪些、砍哪些"
- ❌ 不预设叙事主线
- ❌ 不重组信息结构

**材料理解报告模板**:

```markdown

**主题**: {一句话}

**核心信息**:
1. {观点 1}
2. {观点 2}
3. {观点 3}

**关键数据/论据**:
- {数据或引用 1}
- {数据或引用 2}

**独特价值**: {这份材料里有什么是别的材料没有的}

**信息完整性**: {是否需要补充、是否过时}

---

### 材料 2：{名称}
（同上格式）

---

### 交叉分析

**关联**: 材料 1 和 2 之间的关系（互补 / 重叠 / 冲突）

**各材料独特价值点**:
- 材料 1 独有: ...
- 材料 2 独有: ...
- 两者共通: ...

**组合可能性**:
- 方向 A: 用材料 1 做主线 + 材料 2 做支撑 → 可能的叙事线

**信息缺口**: {有没有需要搜索补充的信息}

---

### 待确认

```

**确认机制**:
- 报告输出后暂停，等用户确认
- 用户确认后 → 进入 Step 1（需求分析，此时才带活动滤镜）

- 用户说"跳过"→ 直接进入 Step 1（用默认理解）

### Step 0.7: E 场景定向重做（局部修改专用）

> E 局部修改是第三优先级的高频轻量场景。本 Step 不走 Step 0.6-5 的完整流程，只定向重做用户指定的页面。

**核心原则**：
- ❌ 不破坏整体叙事结构
- ❌ 不重做所有页面，只重做指定页面
- ✅ 保持风格一致（沿用原 PPT 的视觉规范）
- ✅ 重做后的页面与周围页面逻辑连贯

**执行流程**：

```
输入: 原 PPT + 用户指定"改哪几页" + 新内容要求
│
├─ Step 0.7.1: 识别目标页面
│   ├─ 用户明确指定："改第 3 页" / "替换第 5-7 页"
│   ├─ 用户模糊指定："这个场景太弱了" → AI 识别需重做的页面
│   └─ 输出: 目标页面列表（含页码 + 重做原因）→ 用户确认
│
├─ Step 0.7.2: 分析周围上下文
│   ├─ 前一页的核心观点是什么
│   ├─ 后一页的核心观点是什么
│   ├─ 目标页在整体叙事线中的位置
│   └─ 输出: 上下文摘要 → 确保重做后逻辑连贯
│
├─ Step 0.7.3: 定向叙事设计（仅目标页面）
│   ├─ 用 Layer 1A 普适方法论设计目标页的叙事
│   ├─ 保持与上下一页的转场逻辑
│   └─ 输出: 目标页的故事脚本（一句话主旨 + 详细内容）
│
├─ Step 0.7.4: 交美工局部替换
│   ├─ 告知美工："只替换这 X 页，原有页面不动"
│   ├─ 告知美工："新页面需沿用原 PPT 视觉规范"
│   └─ 输出: 局部替换后的完整 PPT
│
└─ Step 0.7.5: 整体检查（轻量）
    ├─ 新页面是否与原叙事线连贯
    ├─ 新页面是否与周围页面风格一致
    └─ 总时长是否仍在约束范围内
```

**Step 0.7 不输出的内容**：
- ❌ 不输出完整的叙事大纲
- ❌ 不输出完整的页数配额
- ❌ 不做 Story Completeness 评分（局部修改不适用全局评分）

**Step 0.7 输出的内容**：
```markdown
# E 场景局部修改报告

## 目标页面
| 页码 | 原核心观点 | 重做原因 | 新叙事方向 |
|------|-----------|---------|-----------|
| P3 | "AI 能做什么" | 内容过时 | "AI 在XX场景已经能交付" |
| P5 | "技术架构" | 太技术 | 改成业务语言 |

## 上下文摘要
- P2 → **P3** → P4 过渡：P2 讲痛点，P3 讲 AI 能做什么，P4 讲落地路径
- P3 重做后需保持"痛点→方案→路径"的节奏

## 目标页故事脚本
### P3 — 场景深化（60s）
**新叙事方向**: 不是"AI 能做什么"（太大），而是"在XX场景，AI 已经能交付"
**核心观点**: "XX 部门的 XX 场景，是我们第一个能端到端交付的落地"
**转场**: 承 P2 的痛点，启 P4 的落地路径
```

**⚠️ 重要约束**：
- Step 0.7 输出的叙事设计只针对目标页面，不重构全局叙事
- 如果用户后续要求"重做完整叙事"，则回退到 Step 0.6 或 Step 1，从头走完整流程
- 如果目标页的重做涉及叙事线的重大调整（如"改变了演讲的核心主张"），需先征得用户确认，再提示是否要完整重构

### Step 1: 认知诊断（Layer 1B，条件执行）

> 仅在新技术落地场景执行。非技术场景跳过本 Step，直接进入 Step 2。

| 用户表述 | 判断 | 对应叙事策略 |
|---------|------|-------------|
| "能不能全自动" | 社媒极端化 ✅ | 需纠正：强调人在回路 |

| "先在某个部门试试" | 试点期 | 聚焦场景，给 ROI |
| "全公司推" | 规模化期 | 强调组织变革、流程融合 |
| 无明确信息 | 缺省：试点期 | 标准叙事即可 |

**输出**: 写入叙事大纲「认知诊断」章节 + 影响开头钩子设计（社媒极端化→用数据震撼开头；试点期→用场景开头）

## Deliverable: 故事脚本模板（Story Spine）

> 故事脚本是叙事大纲的下一层产物：每页 PPT 一句话主旨，连起来就是完整故事线。

### 设计标准

每页的「一句话」是该页 PPT 的 **主旨（Key Message）**：
- ✅ 足够承载一页 PPT 的全部内容
- ✅ 读完这句话就知道这一页在讲什么
- ✅ 连起来就是完整的故事线
- ❌ 不是旁白（"接下来我们看..."）
- ❌ 不是标题（"场景一"）
- ❌ 不是太单薄的过渡句

```markdown
{一句话主旨}

```

### 页面角色速查

| 角色 | 用途 | 示例 |
|------|------|------|
| **钩子** | 30 秒抓住注意力 | "去年有个同事跟我说了一句话..." |
| **数据震撼** | 用数据建立紧迫感 | "25% 的企业拿到了价值，75% 没有" |
| **核心承诺** | 告诉听众今天能得到什么 | "今天讲怎么让你成为那 25%" |
| **悖论引入** | 引入框架性问题 | "第一个坑：时间悖论" |
| **悖论展开** | 具体化悖论现象 | "开源 3 个月迭代一次..." |
| **悖论总结** | 提炼结论 | "使用成熟服务 > 自建能力" |
| **反转** | 颠覆认知 | "技术本身不创造价值" |
| **类比** | 用熟悉的事解释陌生的事 | "就像电力是通用能力，冰箱是价值载体" |
| **架构可视化** | 展示系统性思考 | 四层架构图 |
| **产品定义** | 一句话定义产品价值 | "不是做 Agent，是做让每个人用自然语言完成工作" |
| **场景开头** | 引入具体场景 | "过去：30 分钟写周报。现在：1 分钟" |
| **场景深化** | 挖掘场景背后的用户价值 | "关键不是快，是自动从 5 个数据源提取" |
| **场景总结** | 提炼共同模式 | "三个场景的共同点：高频 × 高痛 × 可衡量" |
| **诚实边界** | 说明做不到什么 | "财务审批最终金额，必须人拍板" |
| **能力对比** | 能 vs 不能的清晰对比 | "不能碰的 vs 端到端交付的" |
| **信任构建** | 解释为什么诚实反而有用 | "客户选我们不是因为最强，是因为知道边界" |
| **核心重申** | 一句话总结全部演讲 | "不要问 AI 能做什么，问什么场景能省时间" |
| **行动号召** | 告诉听众回去做什么 | "选一个高频场景，让 AI 跑一个月" |
| **收尾** | 降低门槛，结束 | "一个月后再决定要不要扩展" |
| **过渡** | 章节之间的衔接 | "认知对齐了。接下来看看问题出在哪" |

## 反模式检查（禁止出现）

```
❌ "一句话全自动"
❌ "我们的技术最强"
❌ 只谈降本增效（裁员视角）
❌ 回避技术的不确定性
❌ 功能罗列，不讲场景价值
❌ 忽视组织变革因素
❌ 制造幻觉，不实事求是
❌ 行业术语错误（"外行话"）
❌ 风格与演讲者/听众不匹配
❌ 故事脚本的一句话太单薄（只是标题/旁白/过渡）
```

## Trigger

### 情境-旅程激活矩阵

> ●重=核心工作量，●中=必须但不重，○轻=快速过，○无=跳过

| 旅程阶段 | A 改造 | B 创作 | C 美化 | D 翻新 | E 局部 |
|---------|--------|--------|--------|--------|--------|
| ① 理解材料 | ●重 | ○轻 | ●中 | ●重 | ○轻 |
| ② 明确约束 | ●重 | ●重 | ○轻 | ●重 | ○无 |
| ③ 构思叙事 | ●重 | ●重 | ○无 | ●重 | ○无 |
| ④ 内容生产 | ●重 | ●重 | ○轻 | ●中 | ○轻 |
| ⑤ 视觉实现 | ●重 | ●重 | ●重 | ●重 | ●轻 |
| ⑥ 验证交付 | ●中 | ●中 | ○轻 | ●中 | ○轻 |

### Trigger

#### 自动触发

| 情境 | 触发时机 |
|------|----------|
| A 改造 | 用户上传多份材料（PDF/文档/链接） |
| D 翻新 | 用户上传他人/旧 PPT，说"帮我变成我自己的演讲" |
| B 创作 | 用户有明确主题，说"帮我从零设计叙事" |
| E 局部 | 用户说"只改这几页" / "替换第 X 页" / "更新第 Y 页内容" |

#### 手动触发

```
用户说:
- "帮我设计叙事"
- "这个故事怎么讲更好"
- "给我叙事建议"
- "检查叙事完整性"
- "帮我写故事脚本"
- "每页 PPT 用一句话"
- "策划"（角色名触发）
- "帮我做策划"
- "帮我做叙事策划"
- "帮我重构这个演讲"（D 翻新）
- "帮我把这个材料理成演讲"（A 改造）
- "帮我从零设计一个 PPT 的叙事"（B 创作）
- "帮我改第 X 页"（E 局部，指向特定页面）
```

#### 不适用场景（明确拒绝）

```
- "帮我美化这个 PPT"（C 美化 → 建议转美工 Skill）
- "帮我生成一个 PPT"（B 创作，但无主题 → 需要先明确主题）
- "把 Word 转成 PPT"（纯格式转换 → 建议用 iSlide/Canva）
```
